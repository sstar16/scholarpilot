关键约定

· backend 用 volume 挂载 ./backend/app，改 Python 代码 restart 即生效

PhaseRunner 内部循环

每个 phase 执行前检查中断标志 → 检查 skip_if(ctx) → 触发 PRE hook → 执行 → 触发 POST hook → 推 SSE。skip_if 是 commit 1d7918f 新增的声明式跳过接口。

flowchart TD
    Start(["PhaseRunner.run(ctx)"])
    Start --> Loop{"还有 phase?"}
    Loop -->|没有| End(["return ctx"])
    Loop -->|有| AN{"can_interrupt?"}
    AN -->|是| CheckInt{"Redis interrupt 标志?"}
    AN -->|否| Skip
    CheckInt -->|是| Partial["synthesize_partial\n+ deliver"]
    CheckInt -->|否| Skip
    Partial --> Abort(["raise PhaseAborted"])
    Skip["phase.skip_if(ctx)?"] --> SkipDecide{"True?"}
    SkipDecide -->|True| LogSkip["artifacts[name]=None"]
    SkipDecide -->|False| PreHook
    LogSkip --> Loop
    PreHook["fire PRE_NAME hook"]
    PreHook --> SSE1["SSE start progress"]
    SSE1 --> Exec["await execute(ctx)"]
    Exec --> Res{"结果?"}
    Res -->|return dict| Save["artifacts[name] = output"]
    Res -->|raise PhaseSkipped| LogSkip2["artifacts[name]=None"]
    Res -->|raise PhaseAborted| Abort
    Res -->|raise Exception| Crash(["round failed"])
    Save --> PostHook["fire POST_NAME hook"]
    LogSkip2 --> SSE2
    PostHook --> SSE2["SSE end progress"]
    SSE2 --> Loop

    style Abort fill:#fef3c7
    style Crash fill:#fee2e2
    style PreHook fill:#fce7f3
    style PostHook fill:#fce7f3

Answer Now 中断

长检索（30-60s）中点"现在就给我答复"，让 PhaseRunner 在下一个 can_interrupt phase 前停下，partial_synthesizer 用已收集的文档合成简短答案。

    participant FE as ChatPanel
    participant API as search.py
    participant RD as Redis
    participant W as Worker (PhaseRunner)
    participant PS as partial_synthesizer
    participant LLM
    Note over W: 已 P1-P6, 在 P7 FetchPhase
    U->>FE: 点 Answer Now
    FE->>API: POST /rounds/{id}/answer-now
    API->>RD: SET interrupt_flag:{rid} EX 60
    Note over W: P7 即将进入下一 phase
    W->>RD: GET interrupt_flag
    RD-->>W: "1"
    W->>W: docs_so_far = collect (score>rerank>fetch)
    W->>PS: synthesize_partial(docs, stage)
    PS->>LLM: 1 次 "基于这些先答"
    LLM-->>PS: 部分答案
    PS->>API: deliver_partial_answer
    API->>RD: DEL interrupt_flag
    W->>W: raise PhaseAborted
    Note over W: round.status='complete' (不是 failed)
    API-->>FE: SSE: partial_answer

Supervisor vs Workflow

为什么 ScholarPilot 不用 LangGraph 那种"主 LLM agent 路由 + 子 agent 工作"的范式？因为核心流程是强结构 workflow（项目→关键词→检索→反馈），用 LLM 做路由会带来不必要的 token 成本、延迟和不可预测性。Anthropic 自己的 "Building effective agents" 也强调 "Don't use agents when workflows suffice"。

flowchart LR
    subgraph "Supervisor (LangGraph 风格)"
      S1["主 LLM Agent\n(理解+决策+调度)"]
      S1 -->|tool call| W1[Worker A]
      S1 -->|tool call| W2[Worker B]
      S1 -->|tool call| W3[Worker C]
      W1 --> S1
      W2 --> S1
      W3 --> S1
    end
    subgraph "ScholarPilot (workflow + agent)"
      SM["状态机\n(Python 代码)"]
      SM --> SP1["Phase A → Agent A"]
      SP1 --> SP2["Phase B → Agent B"]
      SP2 --> SP3["Phase C → Agent C"]
    end

对比要点

· 路由决策：Supervisor 是 LLM 推理（每条消息 +1 LLM 调用）；ScholarPilot 是 Python switch（确定 + 零成本）

· 输出可预测性：Supervisor 可能跑偏到不该走的路径；ScholarPilot 路径固定

· 适用场景：Supervisor 适合开放助手 / 编码代理；ScholarPilot 适合强结构科研流程
