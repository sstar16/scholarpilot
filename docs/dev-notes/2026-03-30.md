### 环境决策：为什么用两个独立目录

**最初思路**：用 docker-compose override 文件覆盖端口，两套容器跑在同一目录。

**发现的问题**：docker-compose.yml 中 `./backend/app:/app/app` 是 volume 挂载，宿主机目录是同一个。改代码会同时影响两套容器，生产和开发无法真正隔离。

**最终方案**：两个独立 clone 目录。
- `D:\AI\scholarpilot\` — 客户生产环境（不动）
- `D:\AI\scholarpilot-dev\` — Phase 2 开发目录（本仓库）

**卷自动隔离原理**：Docker Compose 默认用目录名作为 project name，卷名格式是 `<项目名>_<卷名>`。两个目录的 pgdata 卷分别叫 `scholarpilot_pgdata` 和 `scholarpilot-dev_pgdata`，天然隔离，数据不会串。

### 事故：在生产目录误执行 git pull

**经过**：在 `D:\AI\scholarpilot`（客户正在用的目录）误执行了 `git pull`，拉入了 Phase 2 的开发代码（21 个文件变更）。

**恢复**：
```bash
git reset --hard ea42c81   # 回退到 pull 前的 commit
docker-compose restart backend worker  # 让旧代码生效
```

**教训**：
- 操作前确认 `pwd` 所在目录
- 生产目录操作 git 前先想三秒
- `git reset --hard` 可以完全回退 fast-forward pull，不会丢失数据

**关键：更新生产版本的 5 种情况**

| 改了什么 | 需要 build？ | 命令 |
|----------|-------------|------|
| Python 代码（app/） | 否 | `restart backend worker beat` |
| requirements.txt | 是 | `build backend worker beat flower && up -d` |
| 前端代码 | 是 | `build frontend && up -d frontend && restart nginx` |
| 数据库 migration | 否 | 先备份，再 `restart backend` |
| .env 环境变量 | 否 | `up -d backend worker beat flower && restart nginx` |

⚠️ 容器重建后 nginx 必须 restart，否则 502（IP 缓存问题）。
