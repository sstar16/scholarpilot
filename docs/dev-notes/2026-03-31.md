## 数据源调研 + 新数据源接入

### 问题背景

截图显示一次真实搜索结果：crossref: 5篇，europe_pmc: 1篇，其余全部为 0 篇。用户三个问题：
1. 各数据源有效性分析
2. 有无其他获取方式（调研 other-projects）
3. 中文优先但全是英文论文的原因

### 调研结论

**数据源失效原因：**
- `pubmed` → 中国网络封锁（TLS 超时）
- `openalex` → 应该能访问，0 篇是本次查询词翻译后不匹配（已加日志便于排查）
- `lens_patent` → 未配置 `LENS_API_TOKEN`（在 lens.org 免费申请）
- `semantic_scholar` → 被 DISABLED_SOURCES 禁用（频繁 429）
- `arxiv` → 中国网络封锁，被 DISABLED_SOURCES 禁用

**中文优先但全是英文的根本原因：**
- "chinese_first" scope 是空占位，中文数据源（万方/百度学术）尚未实现
- 查询词经 LLM 翻译为英文，所有现有数据源都是英文数据库

**other-projects 调研（5个项目）：**
- AutoResearchClaw：OpenAlex/S2/Google Scholar + circuit breaker 模式
- evil-read-arxiv：DBLP（CS 顶会）+ S2 联合搜索
- 无一实现中文学术或专利数据库

## Dev View — 检索 Pipeline 可视化（新分支 feat/dev-view）

### 背景

为了在测试阶段诊断检索质量，需要直观看到：
1. LLM 将用户描述翻译成了哪些关键词
2. 各数据库实际收到的查询词是什么
3. 各数据源返回多少篇、耗时多少、是否报错
4. 后处理（去重→评分）后剩多少篇

### 架构设计

**后端增强（`search_engine.py`）：**
每次 `safe_fetch()` 调用用计时 wrapper 包裹，`source_stats` 由原来的 `{status, count}` 扩展为：
```python
{
    "status": "ok" | "error",
    "count": int,
    "query_sent": str,        # 实际发出的查询词
    "year_from": int,
    "year_to": int,
    "max_requested": int,
    "execution_ms": int,      # 耗时毫秒
    "error": str              # 仅失败时
}
```

**后端增强（`search_tasks.py`）：**
在执行检索前，将 `QueryPlan` 核心信息存入 `SearchRound.search_queries`（该字段原本未使用）：
```python
{
    "base_query": str,           # LLM 翻译后的英文查询词
    "expanded_terms": list,      # 扩展词汇
    "exclude_terms": list,       # 排除词
    "year_from/to": int,
    "language_scope": str,
    "sources_selected": list,
    "max_per_source": int,
    "original_chinese_query": str
}
```

**Schema/API：** `RoundStatusOut` 和 `RoundResultsOut` 新增 `search_queries` 字段，通过现有 `/rounds/{id}/status` 和 `/rounds/{id}/results` 端点返回。

### 前端 Pipeline UI

`frontend/src/views/ProjectView.vue`：
- 顶栏新增 **Dev View** 按钮（点击切换，激活时蓝色高亮）
- 展开 4 步 Pipeline 面板（深色主题 #0d1117 底）：

```
[1] 用户输入       → 项目描述 + 研究领域
       ↓
[2] LLM翻译        → 中文核心词 / 英文查询词 / 扩展词 / 排除词 / 时间范围 / 语言
       ↓ 并行发送
[3] 并行检索       → 表格：数据源 | 实际查询词 | 返回篇数 | 耗时
                     按返回量排序，失败行红底，0篇灰色
       ↓ 去重·补全·评分
[4] 后处理结果     → 原始候选数 → 最终筛选数（漏斗节点）
```

### 注意事项

- Dev View 面板仅在 `awaiting_feedback` 或 `complete` 状态下出现（检索完成后）
- 旧轮次（worker 重启前跑的）无 `search_queries` 数据，Step 2 显示提示"请重启 worker 后新开一轮"
- **使用前必须重启 worker**：`docker-compose restart backend worker`
