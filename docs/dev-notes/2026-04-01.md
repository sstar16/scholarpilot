**`backend/app/workers/search_tasks.py`**
- `query_plan_info` 追加 `english_query_source` / `cn_query_source`

**`frontend/src/views/ProjectView.vue`**

## 画像注入系统重构（三项局限修复）

### 背景

用户画像注入存在三个结构性缺陷：
1. Round 2–5 固定注入 top-5 词，不随画像成熟度递进
2. `preferred_keywords` 中英文混存，中文词注入给英文打分器贡献恒为 0，也没有送到中文数据源
3. 画像词只影响排序（`expanded_terms`），不影响 API 查询词，无法扩大召回

### 变更内容

**`backend/app/services/query_builder.py`**

1. **注入量递进**：新增模块常量 `_PROFILE_INJECT_COUNT = {2:3, 3:5, 4:8}`，round 5+ 默认 10
2. **中英文分流**：用 `_CN_PAT = re.compile(r'[\u4e00-\u9fff]')` 实时检测
   - 英文词 → `extra_en` → `expanded_terms`（排序打分）
   - 中文词 → `extra_zh` → 追加到 `original_chinese_query`（中文数据源实际查询词）
3. **API 召回扩展**：round ≥ 3 且 `extra_en` 非空时，top-2 词存入 `profile_query_extension`
4. `QueryPlan` 新增 `profile_injected_en` / `profile_injected_zh` / `profile_query_extension` 字段

**`backend/app/services/search_engine.py`**
- `execute_search()` 中 query 构建改为三路：中文源用 `original_chinese_query`，英文源 round≥3 追加 `profile_query_extension`，否则用原始 `base_query`

**`backend/app/workers/search_tasks.py`**
- `query_plan_info` 追加三个新字段；`profile_keywords` 上限 5→10

**`frontend/src/views/ProjectView.vue`**
- 画像注入区块拆分为三行：
  - `[排序扩展]` 黄色 tag（英文词，影响打分）
  - `[中文追加]` 绿色 tag（中文词，追加到中文源查询）
  - `[API召回扩展]` 橙色（round 3+ 出现，追加到英文 API 查询词）
- 旧轮次数据自动降级到原有 `profile_keywords` 展示（向后兼容）

## 画像锚词机制（B+C）

### 背景

画像是"文档驱动"的——只从反馈文献里提词。如果 Round 1 召回文献整体质量偏，画像会放大偏差。
项目描述词是更稳定的用户意图锚点，应该作为不衰减的底色，并约束 API 召回扩展的方向。

### 变更内容

**`backend/app/services/query_builder.py`**
- **B（锚词底色）**：`anchor_keywords = base_terms`（来自项目描述 LLM 翻译的词），显式跟踪，始终存在于 `expanded_terms`
- **C（扩展词锚点过滤）**：`profile_query_extension` 优先取 `extra_en ∩ anchor_keywords`（反馈词与描述词的交集），交集为空时降级用纯反馈词
- `QueryPlan` 新增 `anchor_keywords` 字段

**`backend/app/workers/search_tasks.py`**
- `query_plan_info` 追加 `anchor_keywords`

**`frontend/src/views/ProjectView.vue`**
- Step 2 新增"**描述锚词**"行（半透明灰色 tag，始终显示项目描述词）
- API召回扩展旁增加状态标签：绿色 `描述已确认`（词在交集里）/ 灰色 `画像降级`（纯反馈词降级）
