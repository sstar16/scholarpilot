# Dev Notes 2026-04-02

## Open-Loop Bucket Architecture（Phase 1.8）

### 背景

原系统是固定 5 轮渐进式检索，反馈后自动开下一轮，第 5 轮后自动进入监控。用户无法控制何时停止、何时再开一轮。反馈按轮次存储，跨轮次没有统一的文献管理视图。

本次重构将系统从"固定管线"改为"开放式、用户主导"模式。

### 核心变更

#### 1. 4 桶分类系统（替代按轮次的 Feedback）

**新模型 `DocumentClassification`**（`models/document_classification.py`）：
- 4 个桶：`very_relevant` / `relevant` / `uncertain` / `irrelevant`
- 唯一键 `(user_id, document_id, project_id)`，跨轮次持久化
- 保留 `classified_in_round_id`（记录首次分类来源轮次）
- 保留 `reason` / `positive_signals` / `negative_signals`（Memory Agent 用）

**Migration 0006**：建表 + 从旧 `feedback` 表迁移数据（relevance 2→very_relevant 等）+ `max_rounds` 置 0

#### 2. 移除固定轮次上限

- `project.max_rounds` 默认改为 0（无限制）
- `progressive_search.create_next_round()` 移除 `max_rounds` 守卫
- `search.py` 的 `start_round` / `prepare_round` 移除上限检查
- 用户随时可开新一轮

#### 3. 解耦反馈与轮次推进

- `feedback.py` 移除自动创建下一轮 + 自动激活监控的全部逻辑（约 100 行）
- 新增 `POST /rounds/{rid}/finalize` 端点：用户手动结束轮次，触发 Memory Agent + Profile 更新
- 文档分类通过 `PUT /documents/{did}/classify` 即时完成，不再批量提交

#### 4. 监控解耦

- `activate_monitoring()` 重构：接受显式 `schedule` + `search_config` 参数，不再绑定"第 5 轮"
- 支持 upsert（已有 MonitorJob 则更新，没有则创建）
- 新增监控管理 API：`enable` / `disable` / `get` / `update` / `results`

#### 5. 前端桶系统

| 组件 | 说明 |
|------|------|
| `BucketSidebar.vue` | 替代 RoundTimeline，左侧 4 桶可折叠列表，支持拖拽换桶 |
| `BucketDropZone.vue` | 替代 DocumentCard 底部的 4 个反馈按钮，点击分类 + fly-to 动画 |
| `useFlyTo.ts` | CSS transform 飞入动画（GPU 加速，支持 prefers-reduced-motion） |
| `RoundHistory.vue` | 动态轮次历史列表，替代固定 5 步 timeline |
| `MonitoringPanel.vue` | 监控开关 + 频率配置 + 最近结果 |
| `bucket.ts` store | Pinia store，管理 4 桶状态，乐观更新 |

**ProjectView.vue 重构**：
- 侧边栏：RoundTimeline → BucketSidebar
- 反馈流程：批量 submitFeedback → 逐篇 classify + 手动 finalize
- 完成面板：不再显示"开始第 N 轮"，改为"开始新一轮" + "开启监控"

## 并行 LLM 关键词优化

### 问题

`_llm_batch_optimize()` 用单次 LLM 调用为 10+ 个数据源生成大 JSON，prompt 长、生成慢，15s 超时后全部 fallback 到 heuristic。

### 修复

重构为 `_llm_parallel_optimize()`：
- 每源独立一次轻量 LLM 调用（prompt 只含一个源的规则，12s 超时）
- 同义词生成独立一次调用
- 全部 `asyncio.gather` 并行，总耗时 ≈ 最慢的单源
- 单源失败只影响那一个源

## 去掉传统评分截断

### 问题

`search_engine.py` 的 `select_top_documents` 用 `max_select = max_per_source × 源数` 截断（如 20×5=100），关键词不匹配但语义相关的文献会被过早过滤，Scoring Agent 永远看不到。

### 修复

`max_select=len(all_docs)`，传统评分仍计算 `_relevance_score`（排序参考 + LLM fallback），但不再截断。全量送 Scoring Agent，由 LLM 斩杀线做唯一过滤。

## Lens.org 专利 API 修复

### 问题
旧 token 过期（401），用户重新申请了 Trial token。配置后 API 返回 400：`Unrecognized fields - [title, applicants, inventors, references_cited]`，Lens API v2 字段结构已变。

### 修复
重写 `fetchers/lens.py` 适配 v2 API：
- 搜索方式：`match.title` → `query_string`（全文搜索）
- 标题：`item.title` → `biblio.invention_title[0].text`
- 申请人：`item.applicants` → `biblio.parties.applicants[].extracted_name.value`
- 发明人：`item.inventors` → `biblio.parties.inventors[].extracted_name.value`
- 被引数：`references_cited.patents` → `biblio.cited_by.patent_count`
- 排序：`_score` → `relevance`
- 摘要：Lens Patent API trial 级别不返回 abstract（字段不存在），后续由 AI 从标题推断

### 测试结果
Token 有效，检索正常返回 CN/US/EP/WO/JP/KR 专利。
