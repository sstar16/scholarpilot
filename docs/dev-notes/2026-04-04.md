# Dev Notes 2026-04-04

## LLM Provider Manager 单例重构（Phase 1.9）

### 背景

LLMProviderManager 在代码库中有 12 个独立实例化点，每个都做 `LLMProviderManager() + load_llm_config()`，每次读 Redis。FastAPI 的 `llm.py` 有半成品单例，但其他 API 路由（search.py / feedback.py / ai_assist.py）完全无视，各自 new。导致：配置管理分散、Redis 无缓存、前后端状态不一致。

### 核心改动

**统一入口 `get_llm_manager()`**（`services/core/llm_config_store.py`）：
- 进程级单例 + 60s TTL Redis 缓存
- 首次调用：创建实例 + 从 Redis 加载
- TTL 内：直接返回缓存，零 Redis 开销
- TTL 过期：in-place refresh 到同一实例
- `save_llm_config()` 后同进程立即生效，worker 60s 内自动刷新
- `asyncio.Lock` 防并发 refresh

**消除 12 个实例化点**：
所有代码从 3 行模板变为 1 行：
```python
# Before (12 处重复)
llm_manager = LLMProviderManager(default_ollama_host=settings.ollama_host)
await load_llm_config(llm_manager, settings.redis_url)
# After
llm_manager = await get_llm_manager()
```

**改动文件**（10 个）：
- `llm_config_store.py` — 核心：新增 `get_llm_manager()` + 缓存
- `llm.py` — 删除 `_llm_manager` / `_ensure_loaded()`
- `main.py` — 简化 lifespan
- `search.py` × 2、`ai_assist.py` × 2、`feedback.py` × 2、`search_tasks.py` × 2、`deep_dive_tasks.py` × 1、`gap_finder.py` × 1、`trend_spotter.py` × 1

**Redis key**：`llm:config`（注意不是 `llm:providers`，之前查错过）

### 其他修复

**`prepare_round` db.refresh 崩溃修复**：
- 长时间 LLM 调用后 ORM 对象脱离 session → `db.refresh(round_)` 抛 `InvalidRequestError`
- 修复：用 `select(SearchRound).where(id == round_.id)` 重新查询替代 refresh

**jiekou.ai 配置更新**：
- Base URL: `https://api.jiekou.ai/openai`（OpenAI 兼容格式，不是 `/anthropic`）
- 模型列表更新：claude-sonnet-4-6、gpt-5.4-mini、gemini-3.1-pro-preview、deepseek/deepseek-v3.1
- Settings.vue `el-input-number` step 改为 1，允许任意 max_tokens

### feedback.py N+1 查询优化

原来循环内每个 feedback item 3 次串行 DB 查询（RoundDocument + Document + Feedback）= O(3N)。

改为 3 次批量预加载（`IN` 查询）= O(3)：
```python
valid_doc_ids = batch select RoundDocument.document_id
docs_map = batch select Document by ids
existing_map = batch select Feedback by ids
```
